# 托盘代理菜单禁用态设计

## 目标

优化桌面端托盘菜单中的代理控制项：

- 同时保留 `开启代理` 与 `关闭代理`
- 根据当前代理状态，将不允许操作的菜单项置灰并禁用
- 避免用户在已开启时重复点击 `开启代理`，或在已关闭时重复点击 `关闭代理`

## 现状

当前托盘菜单位于 `desktop/src-tauri/src/main.rs` 的 `build_tray_menu(...)`：

- `开启代理` 和 `关闭代理` 都是普通文本菜单项
- 两项始终可点击
- 点击后统一走 `handle_tray_menu_action(...)`
- 菜单刷新依赖现有 `build_tray_state() -> fetch_proxy_status()`

问题在于：

- 菜单状态与实际代理状态没有形成约束
- 用户可以重复触发无效操作
- 视觉反馈不准确

## 方案对比

### 方案 A：保留两个菜单项，使用原生 disabled 状态控制

优点：
- 完全符合现有交互路径，用户心智不变
- Tauri 原生支持 `MenuItemBuilder.enabled(false)`，可以真实灰显
- 逻辑改动小，只影响菜单构建层

缺点：
- 无明显缺点

### 方案 B：只显示当前可执行的一项

优点：
- 菜单更短

缺点：
- 状态表达不完整，用户看不到另一种操作当前为什么不可用
- 不符合这次需求

### 方案 C：继续都显示，但只在点击处理时忽略无效操作

优点：
- 实现最粗糙

缺点：
- 不会灰显
- 可点击但无效果，体验差

## 选择

采用方案 A。

## 详细设计

### 菜单构建

在 `build_tray_menu(...)` 中：

- 保留 `MENU_PROXY_ENABLE` 与 `MENU_PROXY_DISABLE`
- 使用 `MenuItemBuilder::with_id(...).enabled(...)` 构建这两个菜单项
- 当代理已开启时：
  - `开启代理` disabled
  - `关闭代理` enabled
- 当代理已关闭时：
  - `开启代理` enabled
  - `关闭代理` disabled

### 逻辑抽取

抽一个纯逻辑 helper，例如：

- 输入：`proxy_enabled: bool`
- 输出：两个菜单项的 enabled 状态

这样测试不会绑定 Tauri 菜单内部结构，只验证核心状态映射。

### 事件处理

`handle_tray_menu_action(...)` 保持现有后端调用路径不变。

可选增加一层保护：

- 如果收到与当前状态相矛盾的动作，直接忽略

但这不是必要条件，因为 disabled 菜单项正常情况下不会触发点击。

### 测试

在 `desktop/src-tauri/src/main.rs` 现有单测区新增：

- 代理关闭时，`开启代理` 可用、`关闭代理` 不可用
- 代理开启时，`开启代理` 不可用、`关闭代理` 可用

测试层级只覆盖状态映射 helper，避免 UI 平台差异导致脆弱测试。
