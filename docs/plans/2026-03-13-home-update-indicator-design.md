# 首页更新提示设计

## 目标

在桌面端和浏览器端统一复用现有 GitHub 更新检查能力：

- 应用启动后自动检查一次更新
- 之后按固定间隔定时检查更新
- 当检测到新版本时，在首页顶栏设置按钮旁展示主题色更新图标提示
- 在设置页增加“首页更新提示”开关，默认开启，并持久化保存

## 现状

当前项目已经具备：

- 基于 GitHub `latest.json` 的更新检查能力，封装在 `frontend/src/features/updates/updateService.ts`
- 设置页中的完整更新卡片 `UpdateCard`
- 应用设置的前后端持久化链路 `AppSettings`

当前缺少：

- 首页级别的轻量更新提示
- 定时轮询更新状态
- 控制首页更新提示的设置项

## 方案

### 方案 A：在 `App.tsx` 直接管理首页更新状态

优点：
- 改动集中，首页顶栏挂点就在 `App.tsx`
- 与设置页保存后的 `appSettings` 状态天然联动
- 不需要新增全局状态库

缺点：
- `App.tsx` 会再承担一部分更新轮询逻辑

### 方案 B：提取单独的 `useHomeUpdateIndicator` Hook

优点：
- 结构更清晰，轮询与 UI 解耦
- 更利于测试

缺点：
- 新增抽象层，文件数增加
- 对当前规模来说略重

### 方案 C：让设置页的 `UpdateCard` 状态向上同步到首页

优点：
- 更新检查逻辑看似可以复用

缺点：
- 首页提示依赖设置页生命周期，设置页未打开时无法工作
- 数据流反直觉，不适合作为常驻提示来源

## 选择

采用方案 A，并保留最小抽象：

- 首页更新状态放在 `App.tsx`
- 更新检查细节继续复用 `createDesktopUpdateService`
- 为 UI 部分提取一个轻量的首页提示组件，避免 `App.tsx` JSX 膨胀

## 详细设计

### 数据模型

新增设置字段：

- `show_home_update_indicator: boolean`

默认值：

- `true`

前后端都纳入 `AppSettings`，通过现有 `/settings/app` 读写链路持久化。

### 更新检查策略

- 应用启动完成且 `appSettings` 可用后，立即静默检查一次
- 之后每 6 小时轮询一次
- 如果设置项关闭：
  - 不显示首页提示
  - 不启动首页轮询定时器
- 如果检查失败：
  - 不弹错误 toast
  - 保持首页静默，避免打扰
- 如果发现新版本：
  - 在首页顶栏设置按钮旁显示主题色图标
  - 图标可点击，打开设置页，方便用户进入完整更新卡片查看和安装

### 首页交互

首页顶栏右侧结构调整为：

- 设置按钮
- 更新提示按钮（仅在有新版本且开关开启时显示）
- 代理开关
- 添加账户按钮

视觉要求：

- 使用主题色 `#3e5be8`
- 使用图标提示，不加大面积文案
- 与设置按钮并排，风格极简

### 设置页交互

在现有桌面偏好或通用设置区域新增：

- 标题：`首页更新提示`
- 描述：`定时检查 GitHub 新版本，并在首页顶栏显示更新图标提示。`
- 默认开启

保存设置时随现有设置一起提交。

### 测试

后端：
- `AppSettings` 默认值包含新字段
- `SaveAppSettings/GetAppSettings` 能正确存取新字段

前端：
- 首页在检测到更新时显示图标
- 关闭开关时不显示图标且不启动定时轮询
- 点击首页更新图标会进入设置页
- 设置页能显示并修改该开关

## 风险与控制

风险：GitHub 更新检查失败导致首页频繁报错。

控制：
- 首页静默检查不弹 toast
- 仅设置页里的显式“检查更新”按钮继续展示错误

风险：首页与设置页重复创建更新服务，导致额外网络请求。

控制：
- 首页轮询频率固定 6 小时
- 设置页仍保持显式更新卡片逻辑，不强行共享瞬时状态，避免过度耦合
